# TraceMind 技术改造路线图

## 1. 目标

TraceMind 当前已经具备多模态 RAG 问答能力，但整体形态仍更接近单轮问答 demo。  
如果要进一步演进为面向真实业务场景的可信产品客服 Agent，后续改造应重点围绕以下三个目标展开：

1. 从单轮问答升级为多轮澄清式问答。
2. 从纯模型分类升级为基于证据的路由与检索。
3. 从可运行原型升级为可承压、可治理、可持续演进的在线系统。

---

## 2. 改造原则

后续技术改造建议遵循以下原则：

1. 优先优化真实效果，再补齐工程稳定性，最后做长期抽象。
2. 优先做对现有主链路侵入较小、收益明显的改造。
3. 所有新增能力都尽量模块化，避免把路由、检索、模型调用和业务逻辑继续耦合在一起。
4. 所有关键决策点都尽量保留依据、日志与兜底行为，支撑“可信”定位。

---

## 3. 可执行改造清单

## P0：优先解决“能不能答对”

### 3.1 模糊问题识别与追问机制

**目标**  
解决真实客服场景中用户首轮提问模糊、信息不足、无法直接检索的问题。

**建议改造**

1. 在主链路前增加 `clarifier` 节点。
2. 判断当前问题是否具备直接回答所需的信息。
3. 对于信息不足的问题，不直接进入检索，而是输出：
   - 一句最小必要追问；
   - 或 3 到 5 个候选问题/候选意图供用户选择。
4. 对“只发截图、只说不好用、只说办不了”等典型模糊输入进行专项处理。

**建议落地**

- 新增文件：`tracemind/clarifier.py`
- 在 [tracemind/pipeline.py](D:/找实习/TraceMind/tracemind/pipeline.py) 中插入澄清分支
- 为追问结果定义统一结构，例如：
  - `status`
  - `need_clarification`
  - `clarifying_question`
  - `candidate_intents`

**验收标准**

1. 至少覆盖以下 4 类模糊问题：
   - 功能不会用
   - 页面报错
   - 业务不会办
   - 用户只上传图片
2. 模糊问题不再直接落入空召回或低质量回答。

---

### 3.2 单轮问答升级为多轮会话

**目标**  
让用户补充的信息能够进入后续路由、检索与回答流程，而不是每轮都当成独立问题。

**建议改造**

1. 在接口层引入 `session_id`。
2. 保存最近几轮会话的结构化上下文。
3. 区分：
   - 当前问题
   - 追问补充信息
   - 历史约束条件
4. 在进入检索前做一次问题重写，将当前轮与历史补充信息融合为更明确的问题。

**建议落地**

- API 层新增 `session_id`
- 新增会话状态对象，例如 `conversation_state`
- 在 [tracemind/api.py](D:/找实习/TraceMind/tracemind/api.py) 和 [tracemind/pipeline.py](D:/找实习/TraceMind/tracemind/pipeline.py) 之间传递上下文

**验收标准**

1. 用户在第二轮补充“我是 iPhone”“我是企业账号”“我在支付页面”等信息后，后续回答明显收敛。
2. 多轮场景下检索召回质量高于单轮裸检索。

---

### 3.3 证据驱动路由替代纯 LLM 分类

**目标**  
避免模型在复杂业务和系统场景下凭基座知识硬猜业务类型、产品类型或知识来源。

**建议改造**

1. 在正式路由前增加轻量级粗召回。
2. 先召回候选知识摘要，再让模型基于候选证据做路由判断。
3. 路由输入不再只有用户问题，而是：
   - 用户问题
   - 候选产品/系统摘要
   - 候选 FAQ 主题
   - 候选手册目录或章节摘要
4. 路由结果附带依据，便于后续调试和治理。

**建议落地**

- 新增 `router_retriever`
- 对现有查询分类逻辑做拆分：语言识别、业务域粗筛、产品范围缩小、知识源选择
- 优先复用 `catalog/` 下已有目录与摘要结果做路由辅助

**验收标准**

1. 路由结果可解释，能够输出“命中的候选依据”。
2. 对复杂业务问题的错路由率低于当前方案。

---

## P1：补齐客服系统的稳定性和兜底能力

### 3.4 增加统一降级与兜底机制

**目标**  
让最差结果不再是系统报错或幻觉回答，而是可控、保守、可恢复的服务行为。

**建议改造**

定义统一降级层级：

1. `L1`：正常多模态 RAG 回答
2. `L2`：返回知识片段 + 保守总结
3. `L3`：提示用户补充关键信息
4. `L4`：建议转人工或生成工单

触发条件建议包括：

- 检索为空
- 召回片段不一致
- 模型超时
- 模型异常
- 低置信回答

**建议落地**

- 新增 `fallback_manager`
- 在 [tracemind/answer_product_query.py](D:/找实习/TraceMind/tracemind/answer_product_query.py) 和主链路之间增加统一兜底控制

**验收标准**

1. 出现异常时系统仍能返回可读响应。
2. 用户不会直接看到内部错误堆栈或生硬报错。

---

### 3.5 模型与外部依赖的超时、重试、熔断

**目标**  
避免单个模型服务、Embedding 服务、Milvus 或外部依赖异常时拖垮整条调用链。

**建议改造**

对以下依赖统一封装：

1. LLM 调用
2. Embedding 调用
3. 向量数据库调用
4. 未来可能接入的业务系统接口调用

每一类调用统一支持：

- timeout
- retry
- fallback
- circuit breaker

**建议落地**

- 新增统一调用层，例如 `tracemind/model_client.py`
- 对外暴露稳定接口，屏蔽不同模型和供应商差异

**验收标准**

1. 外部依赖异常时，系统能快速失败并走降级链路。
2. 系统不会因为单次调用卡死而长时间阻塞。

---

### 3.6 接口异步化与并发控制

**目标**  
让系统具备面向多用户并发访问的基础能力，而不是只能服务低并发 demo 负载。

**建议改造**

1. 将 FastAPI 接口逐步改为 async。
2. 对高 IO 环节逐步做异步化改造。
3. 引入并发控制，例如：
   - semaphore
   - 请求队列
   - 限流策略
4. 为后续横向扩展预留接口。

**建议落地**

- 优先改造 [tracemind/api.py](D:/找实习/TraceMind/tracemind/api.py)
- 梳理模型调用与检索调用中的阻塞点

**验收标准**

1. 多请求同时进入时不会完全串行阻塞。
2. 在高 IO 场景下系统平均响应时间更稳定。

---

## P2：做长期可演进的工程抽象

### 3.7 建立 Prompt Registry

**目标**  
解决提示词散落在代码中、难以维护、难以版本管理和评测的问题。

**建议改造**

按职责拆分 prompt，例如：

1. 路由 prompt
2. 追问 prompt
3. 检索回答 prompt
4. refine prompt
5. 降级回复 prompt

每个 prompt 至少记录：

- 名称
- 版本号
- 适用场景
- 输入变量
- 示例

**建议落地**

- 新建 `tracemind/prompts/` 目录，或建立 `prompts_registry.py`
- 回答日志中记录具体使用的 prompt 版本

**验收标准**

1. 任意一次回答都能追溯所用 prompt 版本。
2. prompt 更新不再依赖散点式手工修改。

---

### 3.8 建立统一模型抽象层

**目标**  
提高模型可替换性，减少未来更换模型、接入新供应商时的重构成本。

**建议改造**

对以下能力统一抽象：

1. chat model
2. multimodal model
3. embedding model
4. rerank model

统一屏蔽：

- 参数命名差异
- 供应商 SDK 差异
- 重试和超时策略
- 结构化输出差异

**建议落地**

- 新建 `tracemind/models/`
- 将业务层与具体模型 SDK 解耦

**验收标准**

1. 更换底层模型时，主 pipeline 基本不需要重写。
2. 多模型 A/B 对比实验更容易实施。

---

### 3.9 知识库与路由配置分层

**目标**  
减少新增业务域、知识导入、检索优化时的大幅重构。

**建议改造**

将知识组织为多层结构：

1. 业务域层
2. 产品/系统层
3. 文档块层
4. FAQ/案例层

并尽量让以下内容配置化：

- 哪些知识库参与召回
- 哪些问题优先走 FAQ
- 哪些问题必须追问
- 哪些问题必须转人工

**建议落地**

- 调整 metadata schema
- 检索逻辑支持按层过滤
- 将部分路由规则从代码中抽离到配置层

**验收标准**

1. 新增一个业务域时，不需要重写主检索链路。
2. 路由与知识演进对主流程的冲击明显降低。

---

## P3：补齐“可信 Agent”的治理闭环

### 3.10 增加证据展示与置信信号

**目标**  
让系统回答具备可追溯性和可信感，而不是只给结论。

**建议改造**

1. 回答中携带引用片段或来源标识。
2. 对低置信结果输出保守话术。
3. 必要时明确提示“当前依据不足，需要补充信息”。

**验收标准**

1. 用户能区分“有依据回答”和“待补充回答”。
2. 项目能更好支撑“可信产品客服 Agent”定位。

---

### 3.11 增加链路观测与问题回放

**目标**  
让系统可运维、可排障、可持续优化。

**建议改造**

记录每次请求的关键链路信息：

- 路由结果
- 检索命中
- 使用的 prompt 版本
- 使用的模型
- 耗时
- fallback 触发情况

**建议落地**

- 增加结构化日志
- 为典型失败案例保留可回放样本

**验收标准**

1. 能定位坏案例是路由问题、检索问题还是生成问题。
2. 能对优化前后效果做横向比较。

---

### 3.12 建立评测集与回归测试

**目标**  
让后续优化不再只依赖主观感觉，而是有持续可重复的质量验证机制。

**建议改造**

建立专项测试样例，至少覆盖：

1. 模糊问题
2. 多轮追问
3. 图片相关问题
4. 错路由问题
5. 空召回问题
6. 降级与兜底问题

**建议落地**

- 在 `data/` 或 `artifacts/` 下单独维护评测数据
- 为关键链路增加回归脚本

**验收标准**

1. 每次改造后都能快速回归。
2. 能量化回答质量、稳定性和路由准确率变化。

---

## 4. 推荐实施顺序

如果只优先推进最值得做的 3 项，建议顺序如下：

1. 模糊问题识别与追问机制
2. 多轮会话状态管理
3. 证据驱动路由

这三项完成后，TraceMind 会从单轮问答系统明显升级为更贴近真实客服工作流的 Agent。

如果继续推进下一批高价值改造，建议依次增加：

4. 统一降级与兜底机制
5. 模型与外部依赖的统一调用封装

---

## 5. 阶段性路线图

### 第一阶段：会追问

- 解决模糊提问
- 引入多轮上下文
- 让用户补充信息真正参与回答

### 第二阶段：会基于证据路由

- 路由前置粗召回
- 基于候选知识做决策
- 缩小检索范围，减少误判

### 第三阶段：高压和异常下也能稳住

- 异步化
- 并发控制
- 超时、重试、熔断
- 统一降级

### 第四阶段：可治理、可演进、可替换

- Prompt Registry
- 模型抽象层
- 配置化路由
- 评测与回归

---

## 6. 总结

TraceMind 后续最重要的升级方向，不是单纯继续堆模型能力，而是把系统从“能回答”推进到“会澄清、会路由、会兜底、可治理”。

围绕上述路线逐步改造后，TraceMind 将更符合以下定位：

**TraceMind：基于多模态 RAG 的可信产品客服 Agent**
